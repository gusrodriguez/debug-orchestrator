# Architecture

## Overview

The debug orchestrator is a cross-repo debugging system for Claude Code. It connects a
frontend repository to one or more backend repositories, enabling an AI agent to
diagnose and fix bugs that span both sides.

## Three-layer architecture

```
┌─────────────────────────────────────────────────┐
│  Orchestrator (commands/start.md)               │
│  Calls MCP tools, spawns sub-agents, applies    │
│  fixes. Generic — works with any project.       │
├─────────────────────────────────────────────────┤
│  MCP Server (scripts/mcp-server.ts)             │
│  Reads .code-index/, exposes 5 search tools.    │
│  Generic — works with any builder output.       │
├─────────────────────────────────────────────────┤
│  Builder (scripts/builders/<framework>.ts)      │
│  Scans source code, produces .code-index/ JSON. │
│  Framework-specific — one per backend type.     │
└─────────────────────────────────────────────────┘
```

| Layer | Generic? | Purpose |
|-------|----------|---------|
| **Builder** | No | Scans backend source code, produces `.code-index/` JSON |
| **MCP server** | Yes | Reads `.code-index/`, exposes 5 search tools to Claude Code |
| **Orchestrator** | Yes | Calls MCP tools, spawns sub-agents, coordinates diagnosis and fixes |

The contract between builder and MCP server is the `.code-index/` output format. Any
builder that produces that format works with the same MCP server and orchestrator.

## Data flow

```
User describes bug
       │
       ▼
Phase 0: Parse bug description, resolve frontend app
       │
       ▼
Phase 1A: Call MCP tools (search_backend_routes, get_backend_endpoint)
          → Backend brief (endpoints, source files, Prisma usage)
       │
Phase 1B: Spawn Scout sub-agent
          → Frontend brief (route, service call, response handling)
       │
       ▼
Phase 2: Present both briefs to user (HARD STOP)
       │
       ▼
Phase 3: Spawn Cartographer → Diagnosis (side, suspect, hypothesis)
       │
       ▼
Phase 4: Present diagnosis to user (HARD STOP)
       │
       ▼
Phase 5: Spawn Architect → Fix plan (waves, exact edits)
       │
       ▼
Phase 6: Present plan to user (HARD STOP)
       │
       ▼
Phase 7: Execute waves → Worker/Fixer/Specialist per step
       │
       ▼
Phase 8: Verification (tsc, lint)
       │
       ▼
Phase 9: Wrap-up and commit
```

## Sub-agents

The orchestrator delegates work to specialized sub-agents. Each agent is a markdown file
in `agents/` with frontmatter declaring its model tier and tools.

| Agent | Model | Role |
|-------|-------|------|
| **Scout** | Sonnet | Reads frontend code, produces compact brief |
| **Cartographer** | Sonnet | Maps code structure, diagnoses root cause |
| **Architect** | Opus | Creates fix plans with exact before→after edits |
| **Worker** | Haiku | Executes plan steps mechanically (zero judgment) |
| **Fixer** | Sonnet | Retries failed steps with drift tolerance |
| **Specialist** | Opus | Last-resort executor with full reasoning |
| **Guide** | Sonnet | Helps create builder scripts for new backends |

### Model tiering

- **Haiku** — cheapest, fastest. Used for mechanical execution where the plan provides
  every detail. The Worker never makes design decisions.
- **Sonnet** — balanced. Used for tasks requiring judgment within a bounded scope:
  analyzing code, diagnosing bugs, handling anchor drift.
- **Opus** — full reasoning. Used for open-ended design (Architect) and complex
  adaptation (Specialist).

### Escalation ladder

When a Worker (Haiku) fails to execute a step:

```
Worker (Haiku) → Fixer (Sonnet) → Specialist (Opus) → Hot-fix (Architect re-spec)
```

Each tier handles progressively more complex failure modes, but no tier redesigns the
plan. Redesign is Architect's job.

## MCP tools

The MCP server exposes 5 read-only tools:

| Tool | Purpose |
|------|---------|
| `backend_overview` | Package info, function count, model count, overview |
| `search_backend_routes` | Search endpoints by name, route, method, keyword |
| `get_backend_endpoint` | Full details of one endpoint (files, Prisma, schema) |
| `search_backend_models` | Search Prisma models by name, field, or table |
| `get_backend_repo_path` | Absolute paths for sub-agents to edit backend files |

MCP tools are session-level — they are available to the main orchestrator but **not** to
sub-agents spawned via the Task tool. This is why the orchestrator calls MCP tools
directly in Phase 1A.

## Configuration

### `debug-config.json` (in host frontend repo, gitignored)

Generated by `/debug-orchestrator/setup`. Contains:

- `debugOrchestratorPath` — absolute path to this repo
- `apps` — frontend app aliases and paths
- `backends` — backend service configs (repo path, package path, builder info)

### `.mcp.json` (in host frontend repo, gitignored)

Generated by setup. Tells Claude Code how to start the MCP server with the correct
env vars pointing to the backend repo and builder script.

## File locations

After setup, files are distributed across three repos:

```
debug-orchestrator/              (this repo)
├── agents/*.md                  (sub-agent definitions)
├── commands/*.md                (slash commands)
├── scripts/mcp-server.ts       (MCP server template)
└── scripts/builders/*.ts        (builder templates)

host-frontend-repo/              (e.g., adg-octopus)
├── .claude/commands/debug-orchestrator → symlink to debug-orchestrator/commands
├── debug-config.json            (generated, gitignored)
├── .mcp.json                    (generated, gitignored)
└── CLAUDE.md                    (route maps, conventions — used by Scout)

backend-repo/                    (e.g., adg-reports)
└── packages/<service>/
    ├── scripts/mcp-server.ts    (copied from template)
    ├── scripts/builders/*.ts    (copied or generated)
    └── .code-index/             (generated by builder, gitignored)
```
